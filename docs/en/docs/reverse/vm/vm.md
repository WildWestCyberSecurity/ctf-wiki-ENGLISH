# Virtual Machine Analysis

For the virtual machine analysis section, we will explain using a simple crackme as an example.

The corresponding `crackme` can be downloaded here: [FuelVM.exe](https://github.com/ctf-wiki/ctf-challenges/blob/master/reverse/vm/fuelvm/FuelVM.exe)

The corresponding `keygenme` can be downloaded here: [fuelvm_keygen.py](https://github.com/ctf-wiki/ctf-challenges/blob/master/reverse/vm/fuelvm/fuelvm_keygen.py)

The corresponding `IDA database` can be downloaded here: [FuelVM.idb](https://github.com/ctf-wiki/ctf-challenges/blob/master/reverse/vm/fuelvm/FuelVM.idb)

The author of this challenge designed a simple virtual machine with multiple instructions. We will use IDA for analysis. For the sake of explanation, I have renamed some variables in the disassembled output.

## Running the Program

We run the program FuelVM.exe. The interface is shown below:

![start.png](./figure/start.png)

In this interface, we can see two input fields — one for entering the username (Name) and another for entering the key (Key). There are also two buttons: Go for submitting the input, and Exit for quitting the program.

## Obtaining User Input

So we can start from here. To obtain user input, the program needs to call the API `GetDlgItemTextA()`:

``` c
UINT GetDlgItemTextA(
  HWND  hDlg,
  int   nIDDlgItem,
  LPSTR lpString,
  int   cchMax
);
```

The obtained input string will be stored in `lpString`. So we can open IDA and find the location with cross-references to `GetDlgItemTextA()`.

``` asm
.text:00401142                 push    0Ch             ; cchMax
.text:00401144                 push    offset inputName ; lpString
.text:00401149                 push    3F8h            ; nIDDlgItem
.text:0040114E                 push    [ebp+hWnd]      ; hDlg
.text:00401151                 call    GetDlgItemTextA
.text:00401156                 push    0Ch             ; cchMax
.text:00401158                 push    offset inputKey ; lpString
.text:0040115D                 push    3F9h            ; nIDDlgItem
.text:00401162                 push    [ebp+hWnd]      ; hDlg
.text:00401165                 call    GetDlgItemTextA
.text:0040116A                 mov     var_a, 0
.text:00401171                 call    process_input
.text:00401176                 jmp     short locExit
```

As shown above, IDA shows that `GetDlgItemTextA` is called only here, and it is called twice to obtain `inputName` and `inputKey` respectively. Then a variable is initialized to 0. Since the purpose of this variable is not yet clear, we rename it to `var_a` for now. After that, a function call is made followed by a jmp. Because the code at the jmp destination handles program exit, we can determine that the call above is invoking the function that processes user input. Therefore, we rename the jmp destination to `locExit` and the function to `process_input`.

## Processing User Input

We enter the `process_input` function. This function only performs very simple processing on the input strings.

```c
  result = strlength((int)inputName);
  if ( v1 >= 7 )                                // v1 = length of inputName
  {
    *(_DWORD *)&lenOfName = v1;
    result = strlength((int)inputKey);
    if ( v2 >= 7 )                              // v2 = length of inputKey
    {
      i = 0;
      do
      {
        inputName[i] ^= i;
        ++i;
      }
      while ( i <= *(_DWORD *)&lenOfName );
      unk_4031CE = i;
      dword_4031C8 = dword_4035FF;
      initVM();
      initVM();
      __debugbreak();
      JUMPOUT(*(_DWORD *)&word_4012CE);
    }
  }
  return result;
```

First, there is the `strlength()` function. This function uses `cld; repne scasb; not ecx; dec ecx` to calculate the string length and stores the result in `ecx`. This is basic assembly knowledge, so we won't go into further detail. We rename this function to `strlength`.

```asm
.text:004011C2 arg_0           = dword ptr  8
.text:004011C2
.text:004011C2                 push    ebp
.text:004011C3                 mov     ebp, esp
.text:004011C5                 mov     edi, [ebp+arg_0]
.text:004011C8                 sub     ecx, ecx
.text:004011CA                 sub     al, al
.text:004011CC                 not     ecx
.text:004011CE                 cld
.text:004011CF                 repne scasb
.text:004011D1                 not     ecx
.text:004011D3                 dec     ecx
.text:004011D4                 leave
.text:004011D5                 retn    4
.text:004011D5 strlength       endp
```

In the pseudo-C code generated by IDA, there are `v1` and `v2`, which I have annotated. Looking at the assembly, it compares `ecx` with `7`, and `ecx` is the string length. So we can determine that the input requirement is: *both inputName and inputKey must be at least 7 characters long*.

When both `inputName` and `inputKey` are at least 7 characters long, simple transformations are applied to the input. The following is a loop:

```c
      i = 0;
      do
      {
        inputName[i] ^= i;
        ++i;
      }
      while ( i <= *(_DWORD *)&lenOfName );
```

The corresponding Python code is:

``` python
def obfuscate(username):
    s = ""
    for i in range(len(username)):
        s += chr(ord(username[i]) ^ i)
    return s
```

After this, the function assigns values to some variables (these are not important, so we'll skip them).

## Registering SEH

```asm
.text:004012B5                 push    offset seh_handler
.text:004012BA                 push    large dword ptr fs:0
.text:004012C1                 mov     large fs:0, esp
.text:004012C8                 call    initVM
.text:004012CD                 int     3               ; Trap to Debugger
```

`initVM` performs some initialization work before the virtual machine starts (essentially assigning initial values to some registers and related components), which we will discuss later. Here we focus on the SEH part. An SEH handler is registered here, with the exception handling function renamed to `seh_handler`, and then `int 3` is used to manually trigger an exception. Similarly, `401330h` is renamed to `seh_handler3`, and `40135Eh` is the last registered exception handling function. We can infer that this is the real main function of the virtual machine, so we rename `40135Eh` to `vm_main`. (For the SEH and anti-debugging parts, you are encouraged to try dynamic debugging on your own to understand them thoroughly.)

## Restoring Stack Balance

We created a `vm_main` function (after renaming, we still need to create the function for IDA to recognize it), then press `F5` and get a failure. The reason for the failure is stack imbalance. Therefore, we can go to the IDA menu item `Options->General` and check `stack pointer` on the right side. This will display the corresponding stack pointer.

```
.text:004017F2 000                 jmp     vm_main
.text:004017F7     ; ---------------------------------------------------------------------------
.text:004017F7 000                 push    0               ; uType
.text:004017F9 004                 push    offset aError   ; "Error"
.text:004017FE 008                 push    offset Text     ; "The key is wrong."
.text:00401803 00C                 push    0               ; hWnd
.text:00401805 010                 call    MessageBoxA
.text:0040180A
.text:0040180A     locret_40180A:                          ; CODE XREF: vm_main+492j
.text:0040180A 000                 leave
.text:0040180B -04                 leave
.text:0040180C -08                 leave
.text:0040180D -0C                 leave
.text:0040180E -10                 leave
.text:0040180F -14                 leave
.text:00401810 -18                 leave
.text:00401811 -1C                 retn
.text:00401811     vm_main         endp ; sp-analysis failed
```

We go to the bottom where the imbalance is shown. The `jmp vm_main` at the top indicates that the virtual machine is executing a loop internally. The `MessageBoxA` call displays the final error message popup. At the `locret_40180A` position, the stack becomes severely imbalanced after multiple `leave` instructions, so we need to manually restore the stack balance.

This is also straightforward. At position `0040180A`, the stack is already balanced (000), so we just need to change this `leave` instruction to `retn`. Like this:

```
.text:0040180A     locret_40180A:                          ; CODE XREF: vm_main+492j
.text:0040180A 000                 retn
.text:0040180B     ; ---------------------------------------------------------------------------
.text:0040180B 004                 leave
.text:0040180C 004                 leave
.text:0040180D 004                 leave
```

Then you will find that `vm_main` can now generate pseudo-C code with F5.

## Virtual Machine Instruction Analysis

To be honest, the analysis part of the virtual machine is a rather tedious restoration process. You need to compare various small sections of operations to determine what kind of instruction it is and which registers are used. In this crackme, the VM performs a `fetch-decode-execute` loop. The `decode` phase gives us the most information — different instructions are distinguished here based on their respective `opcodes` using `if-else if-else` branches. The actual restoration process is not complicated, but it can become somewhat tedious depending on the number of instructions the virtual machine implements.

The final analysis results are as follows:

| opcode | value |
| ------ | ----- |
| push   | 0x0a  |
| pop    | 0x0b  |
| mov    | 0x0c  |
| cmp    | 0x0d  |
| inc    | 0x0e  |
| dec    | 0x0f  |
| and    | 0x1b  |
| or     | 0x1c  |
| xor    | 0x1d  |
| check  | 0xff  |

Now let's look at the analyzed `initVM` function:

```c
int initVM()
{
  int result; // eax@1

  r1 = 0;
  r2 = 0;
  r3 = 0;
  result = (unsigned __int8)inputName[(unsigned __int8)cur_index];
  r4 = (unsigned __int8)inputName[(unsigned __int8)cur_index];
  vm_sp = 0x32;
  vm_pc = 0;
  vm_flags_zf = 0;
  vm_flags_sf = 0;
  ++cur_index;
  return result;
}
```

There are 4 general-purpose registers (`r1/r2/r3/r4`), 1 `sp` pointer and 1 `pc` pointer, and flags `zf` and `sf`. The previously unknown `var_a` has also been renamed to `cur_index`, which points to the index of the character currently being processed in `inputName`.

We won't go into detail about the multiple instructions implemented by the VM. Let's focus on the `check` operation:

```c
int __fastcall check(int a1)
{
  char v1; // al@1
  int result; // eax@4

  v1 = r1;
  if ( (unsigned __int8)r1 < 0x21u )
    v1 = r1 + 0x21;
  LOBYTE(a1) = cur_index;
  if ( v1 == inputKey[a1] )
  {
    if ( (unsigned __int8)cur_index >= (unsigned __int8)lenOfName )
      result = MessageBoxA(0, aGoodJobNowWrit, Caption, 0);
    else
      result = initVM();
  }
  else
  {
    result = MessageBoxA(0, Text, Caption, 0);
  }
  return result;
}
```

If the value in `r1` equals `inputKey[cur_index]`, it will then check whether the entire `inputName` has been verified. If there are no errors and the comparison is complete, a message box saying `Good job! Now write a keygen.` will pop up. Otherwise, it will continue with `initVM` to enter the next iteration. (If there's an error, of course a message box will pop up indicating the error.)

`cur_index` is incremented by 1 in `initVM`. Do you remember that `initVM` was called twice in `process_input`? Because `initVM` is called twice, the first 2 characters of our `inputKey` can be any characters.

```c
      unk_4031CE = i;
      opcode = vm_pc;
      initVM();
      initVM();
      __debugbreak();
      JUMPOUT(*(_DWORD *)&word_4012CE);
```
